[Unit 4 - 인증 (JWT)]

-- 멀티 디바이스 토큰 관리 --

로그인할 때마다 새 토큰 쌍이 발급됨:
  아이폰 로그인  → access_token A + refresh_token A
  맥북 로그인   → access_token B + refresh_token B
  갤럭시 로그인 → access_token C + refresh_token C

refresh_tokens 테이블 (DB):
  id | token      | user_id | is_revoked
  1  | "token_A"  | 1       | false
  2  | "token_B"  | 1       | false
  3  | "token_C"  | 1       | false
  같은 user_id지만 기기마다 row가 별도로 존재.

access_token vs refresh_token 추적 가능 여부:
  access_token:  기기마다 값이 다르지만 DB에 저장하지 않음.
                 서버는 몇 개가 돌아다니는지, 어느 기기 것인지 모름.
                 서명이 유효하면 그냥 통과. → stateless
  refresh_token: 기기마다 값이 다르고 DB에 저장.
                 서버가 유효한 토큰 목록을 직접 관리 가능. → stateful

로그아웃 방식에 따른 차이:
  refresh_token으로 로그아웃: 해당 토큰(기기) 하나만 revoke → 특정 기기만 로그아웃
  access_token으로 로그아웃:  user_id 추출 후 전체 refresh_token revoke → 모든 기기 로그아웃

access_token을 DB에 저장하지 않는 이유:
  저장하면 매 API 요청마다 DB 조회가 필요해짐.
  JWT의 핵심 장점인 stateless(서명만으로 빠른 검증)가 사라짐.
  결국 세션 방식과 다를 게 없어짐.

토큰을 관리하는 테이블: refresh_tokens 하나뿐.
access_token은 테이블 없음. 서버는 서명 검증만 함.

---

-- Refresh Token --

왜 필요한가?

  액세스 토큰만 있으면 딜레마가 생긴다:
    짧게 설정 (15분) → 보안 좋음. 하지만 15분마다 로그아웃되어 UX 최악.
    길게 설정 (30일) → UX 좋음. 하지만 탈취당하면 30일 내내 악용 가능.

  해결책: 역할이 다른 두 토큰을 분리한다.

두 토큰의 역할 분리:

  액세스 토큰 (Access Token):
    수명: 짧음 (15분 ~ 1시간)
    용도: 실제 API 요청 인증. 매 요청마다 헤더에 포함.
    저장: DB에 저장하지 않음 (stateless). 서버가 서명만 검증.
    탈취 피해: 최대 1시간. 만료되면 자동 소멸.

  리프레시 토큰 (Refresh Token):
    수명: 김 (7일 ~ 30일)
    용도: 액세스 토큰 재발급 전용. 재발급 요청 시에만 사용.
    저장: DB에 저장. 서버가 유효성을 직접 관리.
    탈취 피해: 서버에서 즉시 revoke(무효화) 가능.

전체 인증 흐름:

  1. 로그인
     POST /auth/login → access_token + refresh_token 동시 발급
     refresh_token은 DB의 refresh_tokens 테이블에 저장

  2. API 요청 (액세스 토큰 사용)
     Authorization: Bearer <access_token>
     서버는 DB 조회 없이 서명만 검증

  3. 액세스 토큰 만료
     POST /auth/refresh (refresh_token 전송)
     → DB에서 refresh_token 유효성 확인
     → 기존 refresh_token 즉시 revoke
     → 새 access_token + 새 refresh_token 발급 (토큰 로테이션)

  4. 로그아웃
     POST /auth/logout (refresh_token 전송)
     → DB에서 is_revoked = True 로 변경
     → 이후 해당 refresh_token으로 재발급 불가

토큰 로테이션 (Token Rotation):
  /refresh 호출 시 기존 refresh_token을 revoke하고 새 것을 발급.
  이유: refresh_token이 탈취됐을 때 한 번 사용되면 무효화되므로
        원래 사용자가 다음 재발급 시도 시 실패 → 탈취 감지 가능.

DB에 저장하는 이유:
  액세스 토큰은 stateless (DB 불필요). 서명으로만 검증.
  리프레시 토큰은 서버가 직접 무효화해야 하므로 DB 저장 필수.
  DB에 없거나 is_revoked=True이면 즉시 거부.

우리 코드의 refresh_tokens 테이블:
  token       실제 토큰 값 (secrets.token_urlsafe(32)로 생성된 랜덤 문자열)
  user_id     어느 유저의 토큰인지 (FK → users.id)
  expires_at  만료 시각
  is_revoked  로그아웃/로테이션 시 True로 변경

액세스 토큰 vs 리프레시 토큰 생성 방식 차이:
  액세스 토큰: JWT 형식. payload에 email, exp를 담아 서명.
               DB 조회 없이 서명만으로 검증 가능 (stateless).
  리프레시 토큰: 랜덤 문자열 (secrets.token_urlsafe).
                JWT일 필요 없음. 어차피 DB에서 조회해서 검증하기 때문.

-- OAuth2 vs JWT --

OAuth2: 인증 흐름(프로토콜). "토큰을 어떻게 발급받고 전달할 것인가"를 정의.
        대표 사례: 소셜 로그인 (Google, Kakao 등)
          → 외부 서비스(구글)가 토큰을 발급 → 우리 서버가 그 토큰으로 API 호출
        흐름(flow) 자체가 핵심.

JWT:    토큰 형식(포맷). "토큰 안에 무엇을 어떻게 담을 것인가"를 정의.
        OAuth2 흐름에서 발급하는 토큰을 JWT 형식으로 만들 수 있음.
        JWT는 OAuth2 없이도 독립적으로 사용 가능.

관계:
  OAuth2 = 흐름 (어떻게 인증할 것인가)
  JWT    = 형식 (토큰에 무엇을 담을 것인가)
  → 조합해서 쓸 수 있지만, 같은 개념이 아님.

우리 코드의 방식:
  OAuth2 아님. 자체 JWT 인증.
  클라이언트 → POST /login (email+password) → 서버가 JWT 직접 발급
  소셜 로그인 없이 우리 서버가 토큰 생성/검증 모두 담당.

OAuth2PasswordBearer vs HTTPBearer (FastAPI):
  OAuth2PasswordBearer: 이름에 OAuth2가 붙지만 실제로는 Bearer 토큰 추출 유틸리티.
                        Swagger에 OAuth2 로그인 폼(username/password)을 표시함.
                        우리 로그인이 JSON body를 받아서 Swagger 폼과 불일치 발생.
  HTTPBearer:           단순히 Authorization: Bearer <token> 헤더에서 토큰만 추출.
                        Swagger에 토큰 직접 입력 칸을 표시 → 우리 방식에 적합.

-- JWT란? --

JWT (JSON Web Token): 서버가 "이 사람은 로그인한 사용자다"를 증명하는 토큰.
문자열 형태이며 세 부분이 점(.)으로 구분됨:

  eyJhbGciOiJIUzI1NiJ9.eyJzdWIiOiJ1c2VyQGV4YW1wbGUuY29tIn0.abc123
  ─────────────────── ────────────────────────────────────── ──────
       Header                       Payload                  Signature

  Header:    알고리즘 정보 (예: HS256)
  Payload:   담긴 데이터 (예: {"sub": "user@example.com", "exp": 1700000000})
  Signature: Header + Payload를 서버의 비밀키로 서명한 값

JWT 동작 흐름:
  1. 클라이언트가 이메일/비밀번호로 POST /auth/login 요청
  2. 서버가 검증 후 JWT 토큰 발급 → 클라이언트에 응답
  3. 클라이언트가 이후 요청마다 헤더에 토큰 포함
     Authorization: Bearer eyJhbGci...
  4. 서버가 토큰을 검증해서 누구의 요청인지 확인

세션 방식 vs JWT 방식:
  세션: 서버가 로그인 상태를 DB/메모리에 저장. 서버 부담.
  JWT:  서버가 상태를 저장하지 않음 (stateless). 토큰 자체에 정보가 있음.
        토큰이 유효하면 DB 조회 없이도 사용자 식별 가능.

JWT의 sub (subject):
  토큰이 "누구에 관한 것인지"를 담는 표준 필드.
  보통 user_id 또는 email을 넣음.
  payload = {"sub": "user@example.com", "exp": ...}

JWT의 exp (expiration):
  토큰 만료 시각 (Unix timestamp). 만료된 토큰은 자동으로 거부됨.
  짧게 설정할수록 보안에 유리 (탈취당해도 금방 만료).
  일반적으로 액세스 토큰 15분~1시간, 리프레시 토큰 수일~수주.

HS256 알고리즘:
  HMAC-SHA256. 하나의 비밀키로 서명하고 검증하는 대칭 방식.
  서버만 비밀키를 알고 있으므로, 서명된 토큰은 서버가 만든 것이 보장됨.
  비밀키가 유출되면 누구나 유효한 토큰을 만들 수 있으니 관리 필수.

-- app/core/deps.py (인증 의존성) --

oauth2_scheme = OAuth2PasswordBearer(tokenUrl="/api/v1/auth/login"):
  요청 헤더 Authorization: Bearer eyJ... 에서 토큰 문자열을 자동 추출.
  tokenUrl은 /docs UI에서 로그인 버튼이 호출할 엔드포인트 주소. 기능적 영향 없음.
  토큰이 헤더에 없으면 FastAPI가 자동으로 401 반환.

get_current_user 동작 흐름:
  1. oauth2_scheme → 헤더에서 토큰 추출
  2. decode_access_token(token) → email 반환 또는 None (만료/서명오류)
  3. None이면 401
  4. DB에서 email로 유저 조회
  5. 유저 없으면 401 (토큰은 유효하나 유저가 삭제된 경우 대비)
  6. User 객체 반환

사용 방법:
  async def get_me(current_user: User = Depends(get_current_user)):
      return current_user
  Depends(get_current_user)만 선언하면 토큰 검증 + 유저 조회가 자동 처리됨.
  엔드포인트는 결과로 받은 User 객체만 사용.

에러 메시지를 두 경우 모두 동일하게 쓰는 이유:
  "토큰 만료", "유저 없음"을 분리하면 토큰 유효성 여부가 노출됨.
  "유효하지 않은 토큰입니다"로 통일해서 정보 노출 최소화.

-- app/api/v1/auth.py --

POST /register (회원가입):
  1. get_user_by_email로 이메일 중복 확인
  2. 중복이면 400 Bad Request
  3. hash_password로 비밀번호 해싱 후 create_user → UserResponse 반환

POST /login (로그인):
  1. get_user_by_email로 유저 조회
  2. 유저 없거나 verify_password 실패 시 401 Unauthorized
  3. 통과 시 create_access_token으로 JWT 발급 → Token 반환
     {"access_token": "eyJ...", "token_type": "bearer"}

보안 포인트 — 에러 메시지를 하나로 통일하는 이유:
  "이메일이 없습니다" / "비밀번호가 틀렸습니다"로 분리하면
  공격자가 이메일 존재 여부를 알 수 있음 (계정 탐색 공격 가능).
  "이메일 또는 비밀번호가 올바르지 않습니다"로 통일해서 정보 노출 차단.

  if not db_user or not verify_password(...):
    둘 중 하나라도 실패하면 같은 401 반환. 어느 쪽이 실패했는지 알 수 없음.

token_type = "bearer":
  HTTP 인증 표준 (RFC 6750). 클라이언트가 이후 요청 시
  Authorization: Bearer eyJ... 형태로 헤더에 포함.
  "bearer"는 "이 토큰을 가진 사람에게 권한을 준다"는 의미.

-- app/core/security.py 코드 설명 --

  pwd_context = CryptContext(schemes=["bcrypt"], deprecated="auto")

CryptContext (passlib):
  비밀번호 해싱 방법을 설정하는 객체.
  schemes=["bcrypt"]  bcrypt 알고리즘 사용.
  deprecated="auto"   구버전 해시를 자동으로 업그레이드 처리.

bcrypt:
  비밀번호를 단방향 해싱하는 알고리즘. 복호화 불가.
  같은 비밀번호라도 해시할 때마다 다른 값이 나옴 (salt 자동 포함).
  "1234" → "$2b$12$abc..." (매번 다름)
  DB가 유출돼도 원본 비밀번호를 알 수 없음.

  hash_password("1234")         → "$2b$12$abc..."  (저장)
  verify_password("1234", hash) → True              (검증)

왜 비밀번호를 해싱하나?
  DB에 평문("1234")으로 저장하면 DB 유출 시 비밀번호 즉시 노출.
  해시값만 저장하면 유출돼도 원본 비밀번호 복원 불가.
  verify_password는 입력값을 같은 방식으로 해시해서 저장값과 비교.

  create_access_token(subject: str) -> str:
      expire = datetime.now(timezone.utc) + timedelta(minutes=settings.jwt_expire_minutes)
      payload = {"sub": subject, "exp": expire}
      return jwt.encode(payload, settings.jwt_secret_key, algorithm=settings.jwt_algorithm)

create_access_token:
  subject: 토큰에 담을 식별자. 보통 email 또는 str(user_id).
  expire:  현재 UTC 시각 + 만료 시간.
           timezone.utc를 쓰는 이유: 서버 시간대와 무관하게 동일한 기준으로 계산.
  jwt.encode: payload를 비밀키로 서명해서 JWT 문자열 반환.

  decode_access_token(token: str) -> str | None:
      try:
          payload = jwt.decode(token, settings.jwt_secret_key, algorithms=[settings.jwt_algorithm])
          return payload.get("sub")
      except Exception:
          return None

decode_access_token:
  jwt.decode: 토큰을 검증하고 payload를 반환.
              서명 불일치, 만료, 형식 오류 시 예외 발생.
  except Exception → None: 검증 실패 시 None 반환.
    호출부(deps.py)에서 None이면 401 Unauthorized.
  payload.get("sub"): 토큰에 담긴 식별자(email 등) 반환.

Q. algorithms가 리스트인 이유?
  jwt.decode는 여러 알고리즘을 허용 목록으로 받음.
  하나만 쓰더라도 리스트로 전달해야 함. 라이브러리 API 설계.

Q. decode에서 except를 Exception으로 넓게 잡는 게 맞나?
  jose 라이브러리가 만료/서명오류/형식오류마다 다른 예외를 던짐.
  이 함수의 목적은 "유효한 토큰인가"를 bool처럼 판단하는 것이라
  어떤 이유든 실패하면 None으로 통일하는 게 실용적.
  세분화가 필요하면 JWTError, ExpiredSignatureError 등으로 분기 가능.

[Unit 5 - Celery (비동기 작업 큐)]

-- 개념 --

Celery: Python 비동기 작업 큐. 시간이 오래 걸리는 작업을 백그라운드에서 처리.

구성 요소:
  FastAPI       → Task를 큐에 넣음 (발행자)
  Redis         → 작업 메시지를 담는 큐 (브로커)
  Celery Worker → 큐에서 작업을 꺼내 실행 (소비자)
  Flower        → Celery 작업 상태 웹 모니터링 (localhost:5555)

사용 이유:
  이메일 발송, 파일 처리 등 무거운 작업을 API 응답과 분리.
  클라이언트는 task_id만 받고 즉시 응답 → 백그라운드에서 작업 실행.

Redis DB 번호:
  broker:  redis://host:port/0  → 작업 메시지 저장
  backend: redis://host:port/1  → 작업 결과 저장
  /0, /1은 Redis의 논리적 DB 구분. 충돌 방지.

---

-- app/worker.py --

  celery_app = Celery(
      "worker",
      broker=f"redis://{settings.redis_host}:{settings.redis_port}/0",
      backend=f"redis://{settings.redis_host}:{settings.redis_port}/1",
      include=["app.tasks"],
  )

broker:  Celery가 작업을 넣고 꺼내는 큐. Redis 사용.
backend: 작업 결과를 저장하는 곳. task_id로 결과 조회 시 여기서 읽음.
include: Task가 정의된 모듈 경로. worker 시작 시 자동으로 import.

---

-- app/tasks.py --

  @celery_app.task(bind=True, max_retries=3, default_retry_delay=5)
  def process_item(self, item_id: int) -> dict:
      try:
          time.sleep(20)
          return {"item_id": item_id, "status": "processed"}
      except Exception as exc:
          raise self.retry(exc=exc)

@celery_app.task: 일반 함수를 Celery Task로 등록.
bind=True:         self로 task 인스턴스에 접근 가능. retry 호출에 필요.
max_retries=3:     최대 3번 재시도. 초과 시 MaxRetriesExceededError.
default_retry_delay=5: 재시도 간격(초).
self.retry(exc=exc): 실패 시 재시도 예약 후 현재 실행 종료.

Task 상태:
  PENDING  → 대기 중 (또는 존재하지 않는 task_id)
  STARTED  → 실행 중
  SUCCESS  → 완료
  FAILURE  → 실패 (재시도 초과 포함)
  RETRY    → 재시도 대기 중

-- app/api/v1/tasks.py --

  @router.post("/tasks/process-item/{item_id}")
  async def run_process_item(item_id: int):
      task = process_item.delay(item_id)  # 큐에 작업 넣기
      return {"task_id": task.id}         # 즉시 응답

  @router.get("/tasks/{task_id}")
  async def get_task_status(task_id: str):
      result = AsyncResult(task_id)
      return {
          "status": result.status,
          "result": result.result if result.ready() else None,
      }

delay(): Task를 브로커(Redis)에 넣고 즉시 반환. 실행은 worker가 담당.
AsyncResult(task_id): task_id로 작업 상태/결과 조회.
result.ready(): SUCCESS 또는 FAILURE 상태이면 True.

apply_async() vs delay():
  delay(arg1, arg2)                    → 간단한 호출
  apply_async(args=[arg1], countdown=10)  → 옵션 지정 가능 (지연 실행, 만료 등)

Celery worker 코드 변경 시 재시작 필요:
  FastAPI는 --reload로 자동 반영되지만
  Celery worker는 --reload 없음 → Ctrl+C 후 재시작 필요.

[Unit 4 - Redis]

-- Redis 개념 --

정의:
  Remote Dictionary Server. 데이터를 메모리(RAM)에 저장하는 key-value 저장소.
  디스크가 아닌 RAM에 저장하므로 DB보다 10~100배 빠름.

주요 목적 4가지:
  1. 캐시 (가장 많이 쓰는 용도)
     자주 조회되고 잘 안 바뀌는 데이터를 Redis에 저장.
     클라이언트 요청 → Redis 캐시 확인 → 있으면 바로 반환 (DB 조회 없음)
                                         → 없으면 DB 조회 → Redis 저장 → 반환

  2. 세션 / 토큰 저장
     refresh_token, 로그인 세션 등 임시 데이터 저장.
     TTL 설정으로 만료 자동 처리.

  3. 메시지 브로커
     Celery(Unit 5)의 작업 큐로 사용.
     "이 작업 나중에 처리해줘" 메시지를 Redis에 넣으면 worker가 꺼내서 처리.

  4. 실시간 기능
     채팅, 알림, 실시간 랭킹 등. Pub/Sub 기능으로 메시지 발행/구독.

데이터 구조:
  String     → 토큰 저장, 카운터         (우리가 사용 중)
  List       → 최근 본 상품 목록
  Set        → 좋아요 누른 유저 목록
  Hash       → 유저 세션 정보
  Sorted Set → 실시간 랭킹

장점:
  압도적인 속도: 메모리 기반. DB보다 10~100배 빠름.
  TTL:          키마다 만료 시간 설정. 만료 처리 코드 불필요.
  단순한 API:   GET, SET, DEL 등 명령어가 직관적.
  다양한 자료구조: 용도에 맞는 타입 선택 가능.

단점:
  데이터 유실 위험: 메모리 기반이라 서버 재시작 시 데이터 사라짐.
                    (영속성 설정으로 완화 가능하지만 DB만큼 안전하지 않음)
  메모리 비용:     RAM은 디스크보다 비쌈. 모든 데이터 캐싱 시 비용 증가.
  일관성 관리:     DB 데이터가 바뀌면 캐시도 갱신해야 함.
                    안 하면 오래된 데이터(stale data)가 반환됨.

DB vs Redis:
  DB    → 영구 보존이 필요한 데이터 (유저 정보, 주문 내역)
  Redis → 빠른 접근이 필요하거나 임시 데이터 (캐시, 토큰, 세션)
  실무에서는 둘을 함께 사용. DB가 원본, Redis가 복사본(캐시).

-- TTL과 키 네이밍 전략 --

TTL (Time To Live):
  Redis key에 설정하는 만료 시간(초).
  TTL이 지나면 Redis가 자동으로 key를 삭제.
  별도 만료 처리 코드 불필요.

  redis_client.set(key, value, ex=60)   # 60초 후 자동 삭제
  redis_client.set(key, value, ex=604800)  # 7일 후 자동 삭제

TTL 설정 기준:
  refresh_token: jwt_refresh_expire_days * 86400 (만료일과 동일하게)
  API 캐시:      60초 (데이터 변경 빈도에 따라 조정)
  너무 짧으면:   캐시 효과 없음 (매번 DB 조회)
  너무 길면:     오래된 데이터가 반환될 위험

키 네이밍 전략:
  Redis는 모든 데이터가 하나의 공간에 저장됨.
  prefix로 네임스페이스를 구분해서 충돌 방지 및 관리 용이성 확보.

  패턴: "타입:식별자"
    rt:{token}          refresh token
    cache:items         아이템 목록 캐시
    cache:item:{id}     단건 아이템 캐시
    session:{user_id}   유저 세션

  상수로 관리:
    REFRESH_TOKEN_PREFIX = "rt:"
    ITEMS_CACHE_KEY = "cache:items"
    → 문자열 직접 쓰지 않고 상수로 관리해야 오타 방지

캐시 무효화 (Cache Invalidation):
  데이터 변경(생성/수정/삭제) 시 관련 캐시를 삭제하는 것.
  삭제하지 않으면 오래된 데이터(stale data)가 반환됨.
  다음 조회 요청 시 DB에서 새로 읽어 캐시를 다시 채움.

  create_item → delete_cache("cache:items")
  update_item → delete_cache("cache:items")
  delete_item → delete_cache("cache:items")

-- Refresh Token DB → Redis 이전 --

Redis 저장 구조:
  key:   "rt:1_drfiIFqfKSz..."  (prefix + token)
  value: "1"                     (user_id)
  TTL:   604800초 (7일)          → 자동 만료

DB 방식 vs Redis 방식 비교:
  만료 처리:   DB = expires_at 컬럼 직접 비교 / Redis = TTL 자동 처리
  무효화:      DB = is_revoked = True UPDATE  / Redis = key 삭제
  유효성 확인: DB = SELECT + 조건 검사        / Redis = GET (없으면 None)
  속도:        DB = 느림                      / Redis = 매우 빠름

REFRESH_TOKEN_PREFIX = "rt:" 를 쓰는 이유:
  Redis는 모든 데이터가 하나의 공간에 저장됨.
  캐시 key, 세션 key 등과 섞이지 않도록 prefix로 네임스페이스 구분.
  실무에서 key 네이밍: "rt:{token}", "cache:items", "session:{id}" 형태.

app/repositories/refresh_token.py 변경:
  save:   redis_client.set(key, user_id, ex=expire_seconds)
          ex = TTL(초). 설정하면 만료 시 Redis가 자동으로 key 삭제.
  get:    redis_client.get(key) → str(user_id) 또는 None
          None이면 만료됐거나 revoke된 것. 두 경우를 하나로 처리.
  revoke: redis_client.delete(key) → key 삭제 = 무효화

auth.py 변경 사항:
  login:   expires_at 대신 expire_seconds(초) 전달. db 파라미터 불필요.
           86400 = 하루(초). jwt_refresh_expire_days * 86400 = TTL.
  refresh: DB 객체 조회 대신 get_user_id_by_refresh_token으로 user_id만 받음.
           is_revoked, expires_at 검사 없이 None 여부만 확인.
  logout:  db: AsyncSession = Depends(get_db) 파라미터 완전히 제거.
           Redis 작업은 DB 세션이 불필요.

-- app/schemas/item.py — ItemUpdate 스키마 --

  class ItemUpdate(BaseModel):
      name: str | None = None
      price: float | None = None
      is_available: bool | None = None

모든 필드를 Optional(| None = None)로 선언하는 이유:
  PUT 요청에서 일부 필드만 보내도 나머지는 기존 값 유지.
  실무에서는 PUT/PATCH 구분 없이 이 패턴을 가장 많이 씀.

PUT vs PATCH (HTTP 표준 정의):
  PUT   전체 교체. 보내지 않은 필드는 null/기본값으로 덮어씀.
  PATCH 부분 수정. 보낸 필드만 업데이트.

실무 관행:
  소규모 API는 PUT에 Optional 필드 패턴을 써서 PATCH처럼 동작시키는 경우가 많음.
  엄격하게 표준을 따르려면 PATCH 엔드포인트를 별도로 만들면 됨.

-- app/repositories/item.py — update_item / delete_item --

  async def update_item(db: AsyncSession, item_id: int, data: dict) -> Item | None:
      result = await db.execute(select(Item).where(Item.id == item_id))
      item = result.scalars().first()
      if not item:
          return None
      for key, value in data.items():
          setattr(item, key, value)
      await db.commit()
      await db.refresh(item)
      return item

  async def delete_item(db: AsyncSession, item_id: int) -> bool:
      result = await db.execute(select(Item).where(Item.id == item_id))
      item = result.scalars().first()
      if not item:
          return False
      await db.delete(item)
      await db.commit()
      return True

핵심 패턴:

setattr(item, key, value):
  Python 내장 함수. 런타임에 객체 속성을 동적으로 세팅.
  setattr(item, "name", "배") == item.name = "배"
  update_item에 data dict를 루프 돌며 쓰는 이유:
    어떤 필드가 올지 모를 때, 일일이 if문 쓰지 않고 한 번에 처리.

update_item에서 db.add() 없는 이유:
  DB에서 select로 조회한 객체는 세션이 자동으로 추적 중(dirty tracking).
  setattr로 값만 바꾸면 commit() 시 SQLAlchemy가 UPDATE를 자동으로 실행.
  새 객체를 만들 때만 db.add() 필요.

db.delete(item) → await db.commit():
  delete는 예약, commit은 실제 실행. create와 동일한 2단계 패턴.

반환 타입을 bool로 쓰는 이유:
  삭제 성공/실패 여부만 엔드포인트에 알려주면 충분.
  삭제된 객체 데이터를 반환할 필요가 없음.

-- app/api/v1/items.py — PUT / DELETE 엔드포인트 --

  @router.put("/items/{item_id}", response_model=ItemResponse)
  async def update_item(item_id: int, item: ItemUpdate, db: AsyncSession = Depends(get_db)):
      updated = await item_repo.update_item(db, item_id, item.model_dump(exclude_none=True))
      if not updated:
          raise HTTPException(status_code=404, detail="아이템을 찾을 수 없습니다")
      return updated

  @router.delete("/items/{item_id}", status_code=204)
  async def delete_item(item_id: int, db: AsyncSession = Depends(get_db)):
      deleted = await item_repo.delete_item(db, item_id)
      if not deleted:
          raise HTTPException(status_code=404, detail="아이템을 찾을 수 없습니다")

model_dump(exclude_none=True):
  None인 필드를 dict에서 제외하고 반환.
  {"name": "배", "price": None, "is_available": None}
    → {"name": "배"}
  이걸 repo에 넘기면 "배"만 UPDATE, 나머지는 기존 값 유지.
  exclude_none 없이 넘기면 None이 그대로 DB에 써져 데이터 손실.

DELETE의 status_code=204 (No Content):
  HTTP 표준: 삭제 성공 시 응답 body 없음.
  response_model을 지정하지 않는 이유도 동일 (반환할 데이터 없음).
  함수에 return문이 없어도 FastAPI가 204를 자동으로 반환.

POST의 status_code=201 (Created):
  기존 코드는 200이었으나 HTTP 표준은 생성 성공 시 201.
  GET/PUT은 200(기본값), POST(생성)는 201, DELETE는 204.

Q. update_item에서 item not found를 repo가 None 반환으로 알리는 게 최선인가?
  소규모 API에서는 충분하고 간단한 패턴.
  대규모 실무에서는 커스텀 예외를 정의해서 repo에서 직접 raise하기도 함:
    raise ItemNotFoundError(item_id)
  현재 코드처럼 None 반환 후 엔드포인트에서 처리하는 방식도 표준적인 패턴.

Q. delete_item이 bool을 반환하는데, 실패 시 exception을 repo에서 raise하면 안 되나?
  가능. 하지만 repo는 DB 작업만 담당하고,
  HTTP 에러(404, 500 등) 변환은 엔드포인트(컨트롤러)에서 하는 게 계층 분리 원칙.
  repo가 HTTPException을 raise하면 repo가 HTTP를 알게 되는 것 → 계층 오염.

-- app/core/database.py --
DB 연결 설정 파일. 실무 표준 패턴 (SQLAlchemy 공식 문서 기준).

DATABASE_URL:
  SQLAlchemy가 DB 접속 시 쓰는 연결 문자열.
  postgresql+asyncpg://  → PostgreSQL을 asyncpg 드라이버로 비동기 접속

create_async_engine:
  실제 DB 연결을 관리하는 엔진. 커넥션 풀을 내부적으로 관리.
  echo=True   실행되는 SQL을 터미널에 출력 (개발용, 운영에서는 False)

async_sessionmaker:
  DB 세션을 만드는 공장. 세션은 DB와의 대화 단위 (쿼리 실행, 트랜잭션 관리).
  expire_on_commit=False  커밋 후에도 객체 데이터를 메모리에 유지.
                          없으면 커밋 후 객체 접근 시 DB를 다시 조회.

Base (DeclarativeBase):
  모든 DB 모델이 상속받을 베이스 클래스.
  class User(Base), class Item(Base) 형태로 사용.

get_db:
  Depends(get_db)로 엔드포인트에 세션을 주입할 때 쓰는 함수.
  yield를 쓰는 이유: 엔드포인트 실행 후 with 블록이 자동으로 세션을 닫아줌.
  개발자가 직접 session.close() 호출 불필요.

Q. Base(DeclarativeBase)에서 "모든 DB 모델이 상속받는다"는 게 무슨 뜻?
  모델 = DB 테이블을 Python 클래스로 표현한 것.
  Base를 상속받아야 SQLAlchemy가 해당 클래스를 "테이블과 연결된 모델"로 인식.
  안 받으면 그냥 일반 Python 클래스일 뿐.

  예시:
    class User(Base):
        __tablename__ = "users"
        id = Column(Integer, primary_key=True)
        name = Column(String)

  나중에 User, Item, Post 등 테이블마다 클래스를 만드는데, 전부 Base 상속 필요.

Q. Base를 중간에 만드는 이유?
  1. SQLAlchemy 요구사항: ORM 모델은 DeclarativeBase를 상속받아야 인식됨.
     DeclarativeBase를 직접 여러 클래스에 쓰는 게 아니라,
     Base 하나에만 상속시키고 나머지 모델은 Base를 상속.

  2. 공통 컬럼 한 곳에서 관리:
     class Base(DeclarativeBase):
         id: Mapped[int] = mapped_column(primary_key=True)
         created_at: Mapped[datetime] = mapped_column(default=datetime.utcnow)

     → User(Base), Item(Base) 모두 id, created_at을 자동으로 가짐.
       현재 코드는 pass만 있지만, 공통 컬럼이 생기면 여기에 추가.

Q. 세션을 주입한다는 게 무슨 뜻? 모든 API마다 넣어야 하나?
  세션 = DB와 대화하는 연결 객체. 쿼리 실행, 트랜잭션 관리 담당.

  주입 없이 직접 생성 시:
    async def get_items():
        session = AsyncSessionLocal()   # 직접 만들고
        result = await session.execute(...)
        await session.close()           # 직접 닫아야 함
    문제: 에러 발생 시 close() 안 불릴 수 있음. 매 엔드포인트마다 반복.

  Depends(get_db) 주입 시:
    async def get_items(db: AsyncSession = Depends(get_db)):
        result = await db.execute(...)
        # close 불필요 — FastAPI가 자동 처리
    get_db()가 세션을 만들어서 db 파라미터에 넣어줌 (= 주입).
    엔드포인트가 끝나면 자동으로 세션 닫힘.

  모든 API마다 넣나?
    DB를 쓰는 엔드포인트에만 넣으면 됨.
    DB 접근 없는 엔드포인트 (예: /health)에는 불필요.

  컨트롤러 단에서 주입한다는 의미:
    세션을 파라미터에 선언만 하면 FastAPI가 요청마다 새 세션을 만들어서 넘겨줌.
    개발자는 세션 생성/종료 코드 없이 "나는 DB 세션이 필요합니다"라고 선언만 함.

Q. yield란?
  return과 비슷하지만, 함수를 끝내지 않고 값을 넘겨준 뒤 나중에 다시 돌아올 수 있음.

  return: 값을 반환하고 함수 즉시 종료
  yield:  값을 넘겨주고 일시 정지 → 나중에 재개 가능

  get_db에서의 동작:
    1. 세션 생성
    2. yield session → 엔드포인트에 세션 전달, get_db 일시 정지
    3. 엔드포인트 실행 (DB 쿼리 등)
    4. 엔드포인트 종료
    5. get_db 재개 → with 블록 종료 → 세션 자동 닫힘

  return이었으면 5번이 실행되지 않아 세션을 닫을 수 없음.
  yield를 쓰는 함수를 제너레이터(generator)라고 부름.

Q. 어차피 Depends로 주입하는 건데 왜 yield가 필요한가?
  Depends는 엔드포인트 실행 전에 세션을 주입하는 것까지만 함.
  세션은 엔드포인트가 끝난 후에도 닫아야 하는데,
  return이면 get_db가 이미 종료되어 닫을 코드가 없음.

  yield를 쓰면 FastAPI가 엔드포인트 종료 후 get_db를 재개시켜
  with 블록을 빠져나가며 세션을 자동으로 닫음.

  yield = "전달 전 준비" + "전달 후 정리"를 한 함수에서 처리하는 구조.

Q. with 블록 안에서 return을 쓰면 세션을 바로 닫는 거 아닌가?
  맞음. 그게 바로 return을 못 쓰는 이유.
  return 시 실행 순서:
    1. with 블록 진입 → 세션 생성
    2. return session 실행
    3. with 블록 탈출 → 세션 닫힘  ← 이미 닫힘
    4. 닫힌 세션이 엔드포인트에 전달됨 → 에러

  yield 시 실행 순서:
    1. with 블록 진입 → 세션 생성
    2. yield session → 세션 전달, get_db 일시정지 (with 블록은 열린 채 유지)
    3. 엔드포인트 실행 → 정상 (세션 열려있음)
    4. 엔드포인트 종료
    5. get_db 재개 → with 블록 탈출 → 세션 닫힘

  yield가 with 블록을 열린 상태로 붙잡아두는 역할을 함.

Q. return을 못 쓰는 이유가 "get_db 종료"인가, "세션 사용 불가"인가?
  "세션 사용 불가"가 더 정확함.
  return 시 with 블록이 즉시 탈출되면서 세션이 닫힘.
  닫힌 세션이 엔드포인트에 전달되므로, 엔드포인트에서 DB 쿼리 불가.
  get_db 종료 자체가 문제가 아니라, 세션이 닫힌 채 전달되는 것이 문제.

Q. yield로 다시 돌아오는(재개되는) 건 언제?
  엔드포인트가 응답을 반환한 직후.
  FastAPI가 Depends에서 yield를 발견하면
  엔드포인트 완료 후 자동으로 get_db()를 재개시킴.

  순서:
    요청 → get_db 실행 → yield에서 일시정지
    → 엔드포인트 실행 → 엔드포인트 return
    → FastAPI가 get_db 재개 → with 블록 탈출 → 세션 닫힘
    → 응답이 클라이언트에 전송

Q. 세션 전달은 Depends, 세션 닫기는 async with 인가?
  맞음.
  Depends(get_db) → 세션을 엔드포인트에 전달
  async with     → 세션을 안전하게 닫는 보장
  yield          → 이 둘을 연결하는 다리 (with 블록을 열어둔 채로 세션 전달)

Q. 비동기 함수를 사용할 때 await를 꼭 붙여야 하나?
  네. async def 함수는 호출해도 즉시 실행되지 않고 코루틴 객체를 반환.
  await를 붙여야 실제로 실행됨.

    result = item_repo.get_items(db)        # 코루틴 객체 반환 (실행 안 됨)
    result = await item_repo.get_items(db)  # 실제 실행, 결과 반환

  await는 async def 함수 안에서만 사용 가능.
    def get_item():
        result = await ...  # SyntaxError
    async def get_item():
        result = await ...  # 정상

  정리: async def 호출 시 항상 await, await는 항상 async def 안에서만.

  await의 정확한 의미: "이 작업이 끝날 때까지 기다려라"
  단, 블로킹이 아닌 논블로킹으로 기다림.
    블로킹:    DB 쿼리 대기 중 → 다른 요청 처리 불가 (CPU가 멈춰서 기다림)
    논블로킹:  DB 쿼리 대기 중 → 잠깐 자리 비움 → 다른 요청 처리
               → DB 완료 → 돌아와서 계속
  await = "나는 기다릴 테니 그동안 다른 일 해도 돼"라고 이벤트 루프에 알리는 것.

Q. JS처럼 await 없이 호출하고 넘어가는 게 Python에서도 가능한가?
  가능. asyncio.create_task() 사용.

    # await → 끝날 때까지 기다림
    result = await item_repo.get_items(db)

    # create_task → 실행만 시키고 바로 넘어감
    task = asyncio.create_task(item_repo.get_items(db))
    result = await task  # 나중에 결과가 필요할 때

  JS와의 차이:
    JS: 함수 호출만 해도 Promise가 즉시 실행 시작됨
    Python: await나 create_task 없이 호출하면 코루틴 객체만 반환, 실행 안 됨

-- app/schemas/item.py 수정 --

ORM 객체를 Pydantic 스키마로 변환하려면 from_attributes=True 필요.

  class ItemResponse(BaseModel):
      model_config = ConfigDict(from_attributes=True)
      ...

ConfigDict:
  Pydantic 모델의 동작 방식을 설정하는 객체.

  from_attributes=True:
    Pydantic이 dict 외에 ORM 객체의 속성도 읽을 수 있게 허용.
    FastAPI가 response_model 처리 시 내부적으로 model_validate() 호출.
    ORM 객체를 반환할 때 이 옵션 없으면 에러.

    from_attributes=False (기본값): dict만 읽음
    from_attributes=True:          ORM 객체 속성도 읽음

  기타 옵션 (참고):
    str_strip_whitespace=True  문자열 앞뒤 공백 자동 제거
    frozen=True                인스턴스 생성 후 값 변경 불가

-- app/repositories/item.py (DB 쿼리 레이어) --
DB 쿼리를 담당하는 함수 모음. 엔드포인트에서 직접 쿼리 대신 이 함수들을 호출.

주요 함수 설명:
  select(Item)
    SELECT * FROM items 와 동일. SQLAlchemy에서 SQL을 Python으로 표현.

  result.scalars().all()
    쿼리 결과에서 Item 객체 목록을 꺼냄.
    scalars()  튜플이 아닌 Item 객체로 변환
    .all()     전체 목록 반환 / .first() 첫 번째 하나만 반환

  select(Item).where(Item.id == item_id)
    SELECT * FROM items WHERE id = ? 와 동일.

  db.add(item)
    "이 객체를 다음 커밋 때 INSERT해줘"라고 세션 대기열에 예약.
    아직 DB에 저장 안 됨, 메모리에만 있는 상태.

  await db.commit()
    세션 대기열의 변경사항을 실제 DB에 실행 (INSERT/UPDATE/DELETE).

  await db.refresh(item)
    커밋 후 DB에서 최신 데이터를 다시 읽어옴.
    id처럼 DB가 자동 생성한 값을 item 객체에 반영하기 위해 필요.

Q. db.add의 의미?
  세션은 작업 대기열. add는 예약, commit은 실제 실행.

    item = Item(name="사과", price=1000)  # 메모리에만 존재
    db.add(item)                          # INSERT 예약
    await db.commit()                     # 실제 DB에 저장

  add만 하고 commit 안 하면 DB에 아무것도 저장되지 않음.

  분리하는 이유 = 트랜잭션:
    db.add(item1)
    db.add(item2)
    await db.commit()  # 둘 다 한 번에 저장
    중간에 에러 나면 둘 다 취소 → 일부만 저장되는 상황 방지.

Q. db.add는 INSERT를 의미하나?
  정확히는 "변경사항 추적 등록". 상황에 따라 동작이 달라짐.

  새 객체에 add → 커밋 시 INSERT:
    item = Item(name="사과")
    db.add(item)
    await db.commit()  # INSERT INTO items ...

  세션에서 가져온 기존 객체 수정 → 커밋 시 UPDATE (add 불필요):
    item = await db.get(Item, 1)  # 세션이 이미 추적 중
    item.name = "배"
    await db.commit()  # UPDATE items SET name='배' WHERE id=1

  db.delete(객체) → 커밋 시 DELETE

  세션에서 가져온 객체는 자동으로 변경 감지(추적) → add 없이도 commit 시 반영.

-- app/models/item.py (ORM 모델) --
DB 테이블을 Python 클래스로 표현한 것. SQLAlchemy 2.0+ 문법.

코드:
  class Item(Base):
      __tablename__ = "items"
      id: Mapped[int] = mapped_column(primary_key=True)
      name: Mapped[str]
      price: Mapped[float]
      is_available: Mapped[bool] = mapped_column(default=True)

Q. Mapped[str]을 쓰는 이유? 그냥 str과 다른가?
  SQLAlchemy 2.0부터 도입된 현재 표준 문법.

  이전 방식 (1.x):
    name = Column(String)       IDE가 타입을 모름, nullable 별도 지정 필요

  현재 방식 (2.0+):
    name: Mapped[str]           IDE 타입 인식, NOT NULL
    name: Mapped[str | None]    IDE 타입 인식, NULL 허용

  Mapped[str]의 역할:
    1. "이 필드는 DB 컬럼이다"를 SQLAlchemy에 알림
    2. IDE 자동완성/타입 체크 정상 동작
    3. 타입 힌트만으로 NULL 허용 여부 표현

  그냥 str만 쓰면 SQLAlchemy가 DB 컬럼으로 인식하지 못함.

-- Alembic 마이그레이션 --

초기화:
  uv run alembic init -t async alembic
    -t async  비동기 방식 템플릿 사용
  생성 파일:
    alembic.ini          전역 설정 (DB URL, 경로 등)
    alembic/env.py       마이그레이션 실행 환경 설정 (수정 필요)
    alembic/versions/    마이그레이션 파일들이 쌓이는 폴더
    alembic/script.py.mako  마이그레이션 파일 템플릿 (건드릴 필요 없음)

alembic/env.py 수정 내용:
  from app.core.database import Base, DATABASE_URL
  from app.models import item  # 모델 import (없으면 Base.metadata가 비어있음)

  config.set_main_option("sqlalchemy.url", DATABASE_URL)
  target_metadata = Base.metadata  # None → Base.metadata

alembic/env.py 수정 이유:
  1. target_metadata = Base.metadata
     autogenerate 기능을 위해 필요.
     autogenerate: Python 모델을 보고 마이그레이션 파일을 자동 생성하는 기능.
     Base.metadata에 Base를 상속받은 모든 모델 정보가 담겨있음.
     None이면 Alembic이 모델을 모르는 상태 → autogenerate 불가.

  2. config.set_main_option("sqlalchemy.url", DATABASE_URL)
     Alembic이 마이그레이션 실행 시 DB 접속에 쓸 URL을 설정.
     alembic.ini에 직접 쓰면 비밀번호가 파일에 노출되어 git에 올라갈 위험.
     env.py에서 DATABASE_URL(settings에서 .env를 읽어온 값)로 덮어써서 안전하게 처리.

  3. from app.models import item
     모델 파일을 import해야 Base.metadata에 모델이 등록됨.
     import 없으면 Base.metadata가 비어있어 autogenerate가 아무것도 감지 못함.

Q. target_metadata = Base.metadata 로 하면 자동으로 테이블이 생기나?
  아님. 모델을 볼 수 있게 해주는 것뿐, 테이블 생성은 별개.
  실제 반영은 개발자가 명령어를 직접 실행해야만 일어남.
  target_metadata 역할: 모델과 현재 DB를 비교할 때 기준이 되는 메타데이터 지정.

마이그레이션 두 단계:
  1. uv run alembic revision --autogenerate -m "설명"
     모델과 현재 DB를 비교해서 변경사항을 파일로 기록.
     실제 DB는 변경 없음. alembic/versions/ 에 파일 생성.

  2. uv run alembic upgrade head
     기록된 파일을 읽어 실제 DB에 SQL 실행. 이 시점에 테이블이 생성/변경됨.
     head = 가장 최신 버전까지 전부 반영.

  실무에서는:
    revision으로 생성된 파일을 PR에 포함해 코드 리뷰 후,
    스테이징 환경 테스트를 거쳐 운영 DB에 적용.
    autogenerate는 초안이므로 사람이 검토 필수.

  Git 비유:
    revision  = git commit (변경사항 기록)
    upgrade   = git push   (실제 반영)

-- NestJS vs FastAPI 비교 --

Q. NestJS와 FastAPI의 DB 세션 관리 차이?
  NestJS: Repository가 @InjectRepository()로 커넥션을 내부적으로 보유.
          Controller는 DB 세션을 모름.
          DI 컨테이너가 Repository를 싱글턴으로 주입.

  FastAPI: Depends(get_db)로 요청마다 세션을 endpoint에서 생성.
           세션을 파라미터로 Repository까지 흘려내림.
           함수 기반 DI라 클래스 싱글턴 개념이 없음.

  공통점: Controller → Service → Repository 흐름은 동일하게 구성 가능.
  차이점: 세션의 시작점이 다름.
          NestJS  = Repository 내부에서 관리
          FastAPI = Endpoint에서 만들어 아래로 전달

Q. NestJS는 싱글턴, FastAPI는 싱글턴이 아닌가?
  무엇이 싱글턴인지가 다름.

  NestJS 싱글턴: Service, Repository 클래스 인스턴스
                 앱 시작 시 한 번 생성, 모든 요청이 공유
  FastAPI 싱글턴: engine, AsyncSessionLocal (모듈 로드 시 한 번 생성)

  둘 다 DB 세션은 요청마다 새로 생성 (싱글턴 아님)

  FastAPI는 클래스 기반 DI 컨테이너가 없어서
  Service/Repository를 클래스 싱글턴으로 관리하는 개념 자체가 없음.
  함수로 대체하는 구조.

Q. FastAPI는 DI 컨테이너가 없고 모든게 함수인가?
  클래스를 쓸 수 있지만 강제하지 않음.

  NestJS: DI 컨테이너가 프레임워크 핵심.
          @Injectable() 등으로 클래스 간 의존성을 컨테이너가 관리.
          클래스 기반 강제.

  FastAPI: DI 컨테이너 없음.
           Depends(함수)가 DI 역할을 하지만,
           컨테이너 관리가 아니라 요청 시점에 함수를 실행하는 것.
           클래스/함수 둘 다 가능, 강제 없음.

  FastAPI 철학: "자유롭게 짜되, Depends로 의존성만 선언해라"
  NestJS 철학:  "무조건 컨테이너를 통해서 주입해라"

[Unit 1 - FastAPI 기초]

-- FastAPI / uvicorn --
FastAPI: Python 비동기 웹 프레임워크. async/await 기반.
uvicorn: ASGI 서버. FastAPI 공식 문서 기준 서버이며 실무 표준.
          (ASGI 서버 종류: uvicorn, hypercorn, daphne 등이 있으나 실무에서는 uvicorn이 사실상 표준)

WSGI vs ASGI:
  WSGI   요청 하나 처리 중 다른 요청 대기 (동기)
  ASGI   요청 대기 중 다른 요청 처리 가능 (비동기)
  FastAPI는 비동기 프레임워크이므로 ASGI 서버(uvicorn)가 필요

비유: FastAPI = 주방, uvicorn = 홀 직원 (요청을 받아 주방에 전달)

설치:
  uv add fastapi uvicorn

실행:
  uvicorn app.main:app --reload
    app.main   app/main.py 파일
    :app       그 파일 안의 app 변수 (FastAPI 인스턴스)
    --reload   코드 변경 시 자동 재시작 (개발용)

-- Request / Response 스키마 (Pydantic 모델) --
API가 받고 반환하는 데이터 구조를 Pydantic BaseModel로 정의.

파일 위치: app/schemas/item.py

코드:
  class ItemCreate(BaseModel):    # 요청 body 스키마
      name: str
      price: float
      is_available: bool = True   # 기본값 있음 → 요청에 없으면 자동으로 True

  class ItemResponse(BaseModel):  # 응답 스키마
      id: int
      name: str
      price: float
      is_available: bool

Q. 기본값 없는 필드는 자동으로 채워지나?
  반대. 기본값 있는 필드가 자동으로 채워진다.
    기본값 없음 (name, price)     → 요청에 반드시 포함해야 함. 없으면 422 Validation Error
    기본값 있음 (is_available)    → 요청에 없으면 기본값으로 자동 채움

response_model:
  @router.post("/items", response_model=ItemResponse)
  엔드포인트가 반환할 스키마를 명시. 두 가지 역할:
    1. 반환 데이터를 스키마에 맞게 필터링 (불필요한 필드 자동 제거)
    2. /docs에 응답 형태 자동 문서화

model_dump():
  Pydantic 모델 인스턴스를 dict로 변환.
  **item.model_dump() → name=..., price=..., is_available=... 형태로 풀어서 전달

fake_db:
  실제 DB 없이 메모리(리스트)에 임시 저장하는 방식.
  서버 재시작 시 데이터 사라짐. Unit 2에서 실제 DB로 교체.

422 Validation Error:
  요청 데이터가 스키마와 맞지 않을 때 FastAPI가 자동으로 반환하는 에러.
  별도 처리 없이 Pydantic이 자동으로 검증.

-- 전역 예외 핸들러 & 미들웨어 --

전역 예외 핸들러:
  에러 발생 시 모든 엔드포인트에서 일관된 형태로 응답을 내려주는 장치.
  @app.exception_handler(HTTPException)
  async def http_exception_handler(request: Request, exc: HTTPException):
      return JSONResponse(
          status_code=exc.status_code,
          content={"error": exc.detail, "code": exc.status_code},
      )
  HTTPException을 raise하면 이 핸들러가 가로채서 통일된 형태로 응답.

미들웨어:
  모든 요청/응답을 가로채서 공통 처리를 하는 레이어.
  엔드포인트 실행 전/후에 끼어듦.
  @app.middleware("http")
  async def add_process_time(request: Request, call_next):
      start = time.time()
      response = await call_next(request)   # 엔드포인트 실행
      response.headers["X-Process-Time"] = str(time.time() - start)
      return response
  주요 용도: 요청 처리 시간 측정, CORS 설정, 로깅

HTTPException:
  FastAPI에서 HTTP 에러를 발생시키는 방법.
  raise HTTPException(status_code=404, detail="아이템을 찾을 수 없습니다")

Q. X-Process-Time은 표준 헤더인가?
  커스텀 헤더. X- 접두사가 붙은 헤더는 관례적으로 커스텀 헤더를 의미.
  표준 헤더: Content-Type, Authorization, Cache-Control 등
  커스텀 헤더: X- 로 시작하며 서버가 임의로 추가한 헤더

-- 의존성 주입 (Depends) --
공통 로직을 함수로 분리하고 FastAPI가 자동으로 주입하는 패턴.

문제: Depends 없이는 모든 엔드포인트마다 공통 로직을 반복해야 함
  @router.get("/items")
  def get_items():
      token = request.headers.get("Authorization")  # 매번 반복
      user = verify_token(token)                     # 매번 반복
      db = get_db_connection()                       # 매번 반복

해결: Depends로 분리
  @router.get("/items")
  def get_items(user=Depends(get_current_user), db=Depends(get_db)):
      ...  # 공통 로직은 FastAPI가 알아서 실행 후 주입

동작 방식:
  엔드포인트 호출 시 FastAPI가 Depends(함수)를 보고
  해당 함수를 먼저 실행한 뒤 결과를 파라미터에 자동 주입.
  개발자가 직접 함수를 호출할 필요 없음.

주요 사용처:
  DB 세션       db: Session = Depends(get_db)
  현재 유저     user: User = Depends(get_current_user)
  권한 체크     _ = Depends(require_admin)

핵심: 엔드포인트는 "무엇을 할지"만 담당, 공통 준비 작업은 Depends로 분리.

-- 라우터 분리 구조 --
엔드포인트를 기능별 파일로 나누고 한 곳에서 모아 등록하는 패턴.

구조:
  main.py
    └─ api_router (prefix: /api/v1)
         └─ health_router (tag: health)
              └─ GET /health

파일별 역할:
  main.py              FastAPI 앱 생성, 최상위 라우터 등록만
  api/v1/router.py     v1에 속한 모든 라우터를 한 곳에서 모아 등록
  api/v1/health.py     health 관련 엔드포인트만

코드:
  # api/v1/health.py
  from fastapi import APIRouter
  router = APIRouter()

  @router.get("/health")
  def health_check():
      return {"status": "ok"}

  # api/v1/router.py
  from fastapi import APIRouter
  from app.api.v1.health import router as health_router

  api_router = APIRouter()
  api_router.include_router(health_router, tags=["health"])

  # main.py
  from fastapi import FastAPI
  from app.api.v1.router import api_router

  app = FastAPI()
  app.include_router(api_router, prefix="/api/v1")

확장 시:
  api_router.include_router(health_router, tags=["health"])
  api_router.include_router(user_router, tags=["users"])   ← 추가
  api_router.include_router(auth_router, tags=["auth"])    ← 추가
  main.py는 건드릴 필요 없음

tags: /docs에서 엔드포인트를 그룹으로 묶어 보여주는 용도. 기능적 차이 없음.

Q. __pycache__는 왜 생기나?
  Python이 .py 파일을 실행할 때 바이트코드로 컴파일한 결과를 캐싱해두는 폴더.
  다음 실행 시 소스가 바뀌지 않았으면 컴파일을 건너뛰고 캐시를 사용 → 실행 속도 향상.
  개발자가 직접 건드릴 일 없음. git에 올리지 않도록 .gitignore에 추가.
    __pycache__/
    *.pyc

Q. docker compose가 있는데 왜 uvicorn을 따로 실행하나?
  개발 단계에서는 역할을 나눈다.
    docker compose up          PostgreSQL, Redis 등 인프라
    uv run uvicorn ... --reload FastAPI 앱 (로컬 직접 실행)
  앱을 Docker로 띄우면 코드 변경 시마다 이미지를 새로 빌드해야 해서 불편.
  --reload로 로컬에서 실행하면 저장 즉시 반영.
  나중에 배포 단계(Unit 8)에서 Dockerfile을 만들고 docker-compose.yml에 앱도 추가.

[Unit 0 - 개발 환경 세팅]

-- uv --
Python 패키지 매니저. pip + venv + pyenv를 하나로 합친 것.
Rust로 작성되어 기존 도구들보다 훨씬 빠르다.

명령어:
  uv init --no-readme       프로젝트 초기화
  uv python pin 3.12        Python 버전 고정
  uv add <패키지>            의존성 추가 (pyproject.toml에 자동 기록)
  uv add --dev <패키지>      개발용 의존성 추가
  uv run <명령어>            가상환경 안에서 명령어 실행
  uv sync                   pyproject.toml 기준으로 의존성 동기화

생성 파일:
  pyproject.toml    프로젝트 메타데이터 + 의존성 목록
  .python-version   사용할 Python 버전 기록
  uv.lock           패키지 버전 고정 파일 (커밋 O)
  .venv/            가상환경 폴더 (커밋 X, .gitignore에 추가)

-- pyproject.toml --
Python 프로젝트 표준 설정 파일. setup.py, requirements.txt를 하나로 통합.

섹션:
  [project]             프로젝트 이름, 버전, 의존성 목록
  [dependency-groups]   dev 전용 의존성 (테스트, 린터 등)
  [tool.xxx]            각종 도구 설정 (ruff, pytest 등)

uv add 실행 시 dependencies에 자동 추가됨.

-- Ruff --
Python 린터 + 포매터. flake8 + black + isort를 하나로 합친 것.
Rust로 작성되어 매우 빠르다.

설치:
  uv add --dev ruff   (dependency-groups.dev에 추가됨)

pyproject.toml 설정:
  [tool.ruff]
  line-length = 88          한 줄 최대 길이
  target-version = "py312"  대상 Python 버전

  [tool.ruff.lint]
  select = ["E", "F", "I"]
    E: pycodestyle  - 들여쓰기, 공백 등 스타일 규칙
    F: pyflakes     - 미사용 import, 미정의 변수 등 논리 오류
    I: isort        - import 순서 정렬 (표준 → 서드파티 → 로컬)

  [tool.ruff.format]
  quote-style = "double"    큰따옴표 통일
  indent-style = "space"    들여쓰기 스페이스

명령어:
  uv run ruff check .         린트 검사
  uv run ruff check . --fix   자동 수정
  uv run ruff format .        코드 포맷
  uv run ruff format --check . 포맷 변경사항 확인 (실제 수정 없이)

-- .env --
민감한 값(DB 비밀번호, API 키 등)을 코드에서 분리하는 파일.
git에 올리지 않고 환경마다 따로 관리한다. (.gitignore에 추가)

구조:
  # App
  APP_ENV=development
  APP_DEBUG=true
  SECRET_KEY=super-secret-key-change-in-production

  # Database (PostgreSQL)
  DB_HOST=localhost
  DB_PORT=5432
  DB_NAME=pystack_db
  DB_USER=pystack_user
  DB_PASSWORD=pystack_password

  # Redis
  REDIS_HOST=localhost
  REDIS_PORT=6379

  # MinIO (S3-compatible)
  MINIO_ENDPOINT=localhost:9000
  MINIO_ACCESS_KEY=minioadmin
  MINIO_SECRET_KEY=minioadmin
  MINIO_BUCKET=pystack-bucket

-- pydantic-settings --
.env 파일과 환경변수를 Python 클래스로 읽어오는 라이브러리.
BaseSettings를 상속받아 설정 클래스를 만든다.

설치:
  uv add pydantic-settings

파일 위치: app/core/config.py

코드:
  from pydantic_settings import BaseSettings, SettingsConfigDict

  class Settings(BaseSettings):
      model_config = SettingsConfigDict(
          env_file=".env",
          env_file_encoding="utf-8",
      )

      app_env: str = "development"   # 기본값 있음 → .env에 없어도 OK
      app_debug: bool = False
      secret_key: str                # 기본값 없음 → .env에 반드시 있어야 함

      db_host: str = "localhost"
      db_port: int = 5432
      db_name: str
      db_user: str
      db_password: str

      redis_host: str = "localhost"
      redis_port: int = 6379

      minio_endpoint: str
      minio_access_key: str
      minio_secret_key: str
      minio_bucket: str

  settings = Settings()  # type: ignore[call-arg]

핵심 개념:
  - model_config      .env 파일 경로와 인코딩 설정
  - 기본값 없는 필드   앱 시작 시 .env에 없으면 즉시 에러 → 설정 누락을 조기에 잡음
  - 타입 자동 변환    "5432" → 5432, "true" → True (문자열을 각 타입으로 변환)
  - 대소문자 무관      .env의 DB_HOST가 db_host 필드에 자동 매핑
  - 싱글턴 패턴       모듈 하단에 settings = Settings() 하나만 생성,
                      다른 파일에서 from app.core.config import settings 로 사용

-- docker-compose.yml --
여러 컨테이너를 한 번에 정의하고 실행하는 파일.
서비스를 추가할 때마다 여기에 블록을 추가해간다.

구조:
  services:
    db:
      image: postgres:16
      environment:
        POSTGRES_DB: ${DB_NAME}
        POSTGRES_USER: ${DB_USER}
        POSTGRES_PASSWORD: ${DB_PASSWORD}
      ports:
        - "${DB_PORT}:5432"
      volumes:
        - postgres_data:/var/lib/postgresql/data

    redis:
      image: redis:7
      ports:
        - "${REDIS_PORT}:6379"

  volumes:
    postgres_data:

핵심 개념:
  - services          실행할 컨테이너 목록. 나중에 minio, celery 등 추가
  - image             Docker Hub에서 가져올 이미지. 버전 태그 고정 권장
  - environment       컨테이너 안에 주입할 환경변수. ${변수명}으로 .env 값 참조
  - ports             호스트포트:컨테이너포트. 호스트 쪽만 바꾸면 포트 충돌 해결
  - volumes           컨테이너가 꺼져도 데이터 유지. DB처럼 영속성이 필요한 서비스에 사용
  - Redis 볼륨 없음    캐시 용도라 재시작 시 초기화되어도 무방

명령어:
  docker compose up         포그라운드 실행 (로그 실시간 출력)
  docker compose up -d      백그라운드(데몬) 실행
  docker compose ps         실행 중인 컨테이너 상태 확인
  docker compose down       컨테이너 종료
  docker compose down -v    컨테이너 종료 + 볼륨까지 삭제 (DB 초기화)

컨테이너 접속:
  docker compose exec db psql -U pystack_user -d pystack_db
    exec          실행 중인 컨테이너 안에서 명령어 실행
    db            docker-compose.yml에서 정의한 서비스 이름
    psql          PostgreSQL 클라이언트 CLI
    -U            접속할 유저
    -d            접속할 DB 이름

psql 명령어:
  \conninfo     현재 접속 정보 출력 (호스트, 유저, DB명)
  \q            psql 종료

Q. DB 유저와 DB는 언제 만들어지나?
  docker compose up 실행 시 postgres 공식 이미지가 처음 시작될 때
  environment에 넣어준 값을 읽어 자동으로 생성한다.
    POSTGRES_DB       → 이 이름으로 DB 생성
    POSTGRES_USER     → 이 이름으로 유저 생성
    POSTGRES_PASSWORD → 해당 유저 비밀번호 설정
  이미지 안에 초기화 로직이 내장되어 있어 별도 SQL 실행 불필요.
  volumes가 없으면 컨테이너를 내렸다 올릴 때마다 DB가 초기화된다.

Celery + Redis 연결 에러: "nodename nor servname provided"
================================================================

증상:
  uv run celery -A app.worker worker 실행 시 아래 에러 반복
    consumer: Cannot connect to redis://redis:6379/0:
    Error 8 connecting to redis:6379.
    nodename nor servname provided, or not known.

원인:
  .env의 REDIS_HOST=redis 때문.
  'redis'라는 호스트명은 Docker Compose 네트워크 "안에서만" 통한다.
  같은 docker-compose.yml의 서비스끼리는 도커 내부 DNS가
  서비스 이름(redis)으로 서로를 찾게 해준다.
  그런데 celery를 도커가 아닌 "맥북 호스트"에서 직접 실행하면
  호스트는 'redis'가 뭔지 모른다 → DNS 조회 실패.
  Error 8 (EAI_NONAME) = "그런 이름의 호스트 없음".

  worker.py가 settings.redis_host(=.env의 REDIS_HOST)로
  브로커 주소를 만들기 때문에 그대로 redis://redis:6379가 된다.

핵심: celery는 도커로 꼭 떠야 하는 게 아니다.
      어디서 실행하느냐에 따라 REDIS_HOST 값이 달라져야 한다.

  celery 실행 위치          REDIS_HOST 값   이유
  ----------------------    ------------    --------------------------------
  호스트에서 직접 실행      localhost       compose가 ports 6379:6379로
  (uv run celery ...)                       Redis를 호스트 localhost에 열어둠
  도커 안에서 실행          redis           같은 도커 네트워크라
  (celery_worker 서비스)                    서비스 이름으로 접근 가능

주의:
  .env는 호스트 celery와 도커 celery가 공유한다.
  한쪽은 localhost, 다른 쪽은 redis가 필요해서
  값 하나로 둘 다 만족시킬 수 없다. 그래서 보통 하나만 택한다.
    - 개발/디버깅: 호스트에서 직접 (REDIS_HOST=localhost) → 수정·재시작 빠름
    - 배포/통합:   전부 도커로 (REDIS_HOST=redis)

해결 (택1):
  A. 호스트에서 실행
     .env: REDIS_HOST=localhost
     docker compose up -d redis
     uv run celery -A app.worker worker --loglevel=info
  B. 도커로 실행
     .env: REDIS_HOST=redis (그대로)
     docker compose up celery_worker

----------------------------------------------------------------
보충: "호스트에서 실행" 이란?
----------------------------------------------------------------
= celery 워커를 도커 컨테이너가 아니라 내 맥북(호스트 OS)에서
  그냥 하나의 파이썬 프로세스로 직접 띄우는 것.
  (uv run celery -A app.worker worker ... 를 터미널에서 직접 실행)

docker-compose.yml에 celery_worker 서비스가 정의돼 있어도
별개다. 정의만 했다고 자동으로 도는 게 아니라,
docker compose up celery_worker 로 띄워야 컨테이너가 실행된다.
그걸 안 띄우고 터미널 명령어로 돌리는 게 "호스트 실행".

  구분        도커로 실행                    호스트로 실행
  ------      --------------------------     --------------------------
  정의        compose의 celery_worker        없음(그냥 터미널 명령어)
  실행        docker compose up celery_worker  uv run celery -A app.worker worker
  환경        python:3.12-slim 컨테이너 안   맥북 OS 위
  Redis주소   redis (도커 내부 DNS)          localhost (포트 매핑)

개발 단계에서 호스트 실행이 편한 이유:
  - 코드 수정 → Ctrl+C 후 재실행이면 끝(도커는 이미지/설치 과정 거침,
    지금 compose는 매번 pip install uv 까지 다시 함)
  - 로그/디버거 붙이기 쉬움
  - 흔한 조합: 인프라(Redis, DB)는 도커, 앱 코드는 호스트

----------------------------------------------------------------
오해 주의: "docker에서 celery만 빼면 해결?" → 아니다
----------------------------------------------------------------
에러의 원인은 도커의 celery_worker 컨테이너가 아니라,
호스트에서 직접 띄운 celery가 .env의 REDIS_HOST=redis 를
읽어서 실패한 것이다.

  uv run celery ... → .env REDIS_HOST=redis 읽음
                    → 호스트는 redis 못 찾음 → 에러

따라서 docker-compose.yml에서 celery_worker를 지워도
.env가 그대로면 호스트 celery는 똑같이 실패한다.
(선택지를 없앤다고 주소 문제가 풀리는 게 아님)

진짜 해결 = .env 수정:  REDIS_HOST=localhost
  → celery_worker 서비스는 지울 필요 없음.
    docker compose up celery_worker 를 안 하면 어차피 안 뜬다.

celery_worker 삭제가 의미 있는 경우:
  "개발은 호스트에서만 한다"고 정했을 때 헷갈림 방지용 "구성 정리".
  단 이건 에러 해결이 아니라 정리일 뿐.

정리:
  - 에러 해결 필수    → REDIS_HOST=localhost (호스트 실행 시)
  - celery_worker 삭제 → 선택, 에러와 무관

================================================================
.env 수정으로 Celery-Redis 연결 해결 + 전체 구성 개념
================================================================

[발견] .env에서 REDIS_HOST만 혼자 이상했다
  DB_HOST=localhost            ← 호스트용
  MINIO_ENDPOINT=localhost:9000 ← 호스트용
  REDIS_HOST=redis             ← 혼자만 도커용 (문제 원인)
  → 원래부터 앱/celery는 호스트, 인프라만 도커인 구성이었는데
    Redis 주소만 도커용으로 남아 있어서 celery가 죽었다.

[수정] .env 한 줄
  REDIS_HOST=redis  →  REDIS_HOST=localhost
  재실행: uv run celery -A app.worker worker --loglevel=info
  (사전에 docker compose up -d redis 로 Redis 떠 있어야 함)

----------------------------------------------------------------
개념 1) localhost가 되는 이유 = 포트 매핑
----------------------------------------------------------------
  docker-compose redis:
    ports: - "6379:6379"   # 호스트포트:컨테이너포트
  "컨테이너 안 6379를 호스트 6379로 뚫어준다"는 뜻.
  호스트에서 localhost:6379 접속 → 도커가 redis 컨테이너로 전달.

  [맥북 호스트]                       [도커]
  uv run celery
     ↓ localhost:6379
     ──포트매핑(6379:6379)──→ redis 컨테이너 :6379

----------------------------------------------------------------
개념 2) redis(서비스명)가 호스트에서 안 되는 이유
----------------------------------------------------------------
  'redis' 이름은 도커 내부 네트워크 DNS에만 등록돼 있다.
  컨테이너끼리는 이 이름으로 통신하지만 호스트 OS DNS엔 없다.
  → 호스트에서 쓰면 nodename not known 에러.

  접속 주체                     써야 할 주소   이유
  호스트 프로세스(celery/FastAPI)  localhost    포트 매핑으로 뚫린 길
  도커 컨테이너(celery_worker)     redis        도커 내부 DNS

----------------------------------------------------------------
개념 3) 현재 전체 구성 (개발 단계 표준 조합)
----------------------------------------------------------------
  [ 호스트(맥북)에서 직접 실행 ]
    FastAPI  (uv run uvicorn ...)
    Celery   (uv run celery ...)
        │ 둘 다 localhost:포트 로 접속
        ▼
  [ 도커로 실행 (인프라) ]
    PostgreSQL  localhost:5433
    Redis       localhost:6379
    MinIO       localhost:9000

  앱 코드는 수정/재시작 잦으니 호스트, 인프라는 도커.

----------------------------------------------------------------
개념 4) docker-compose의 celery_worker / flower
----------------------------------------------------------------
  지금 구성에선 안 쓰는 잉여 서비스. up에 걸지만 않으면 안 뜸.
  나중에 도커로 통합 실행하려면 그땐 .env를 다시 redis로 바꿔야 함.
  (localhost와 redis를 동시에 만족하는 값은 없음)

  flower(celery 모니터링 UI)도 호스트에서 쓰려면:
    uv run celery -A app.worker flower --port=5555

================================================================
전체 기동 순서 (모두 꺼진 상태에서, 포그라운드/데몬X)
================================================================
터미널 3개를 각각 포그라운드로 띄운다 (-d 안 씀, 로그 실시간 확인).

  터미널 1 - 도커 인프라(DB, Redis)
    docker compose up db redis
    ※ celery_worker, flower 는 빼야 함.
      .env가 localhost라서 도커 안 celery는 redis 못 찾아 죽음.
      (celery/flower는 호스트에서 직접 띄울 것)

  터미널 2 - FastAPI
    uv run uvicorn app.main:app --reload

  터미널 3 - Celery
    uv run celery -A app.worker worker --loglevel=info

포인트:
  - docker compose up 에 -d 안 붙이면 로그가 그 터미널에 흐르고
    터미널이 그 프로세스에 잡힌다. 끄려면 Ctrl+C.
  - FastAPI/Celery는 계속 떠 있는 프로세스라 각자 새 터미널 필요.
  - 순서: 도커(1번) 먼저 → 붙으면 2·3번. 2와 3은 순서 무관.
  - 종료: 각 터미널에서 Ctrl+C.

----------------------------------------------------------------
정리: docker-compose에서 celery_worker, flower 삭제
----------------------------------------------------------------
.env를 localhost로 고친 뒤, celery/flower는 호스트에서 띄우기로
했으므로 docker-compose의 celery_worker, flower 서비스는 잉여.
→ 아예 삭제. 이제 compose엔 db, redis만 남음.

(주의 - 시점 차이)
  - .env가 redis였을 때: 도커에서 빼도 에러 원인(호스트 celery)이
    그대로라 "삭제만으로는 해결 안 됨"이었다.
  - .env를 localhost로 고친 뒤: 에러는 이미 해결됐고, 남은 도커
    celery/flower는 쓸모없는 잉여라 "삭제가 올바른 정리"가 된다.

기동이 단순해짐:
  터미널1: docker compose up         (db, redis만 남아 서비스명 불필요)
  터미널2: uv run uvicorn app.main:app --reload
  터미널3: uv run celery -A app.worker worker --loglevel=info
  (flower) uv run celery -A app.worker flower --port=5555

----------------------------------------------------------------
오해 정정: "도커 celery + 호스트 celery 충돌이 원인?" → 아니다
----------------------------------------------------------------
에러 메시지:
  consumer: Cannot connect to redis://redis:6379/0:
  Error 8 ... nodename nor servname provided, or not known.

이건 "다른 celery와 충돌(conflict)"이 아니라
"호스트 celery가 Redis 주소(redis)를 못 찾은 연결 실패(DNS)".
  - 호스트 celery가 .env의 REDIS_HOST=redis 읽음
  - 호스트 OS엔 'redis' 이름 없음(도커 내부 전용)
  - → 접속 시도조차 실패
도커에 celery가 떠 있든 없든 이 에러는 똑같이 난다.

중요: celery 워커는 여러 개 띄워도 충돌 안 난다(오히려 정상).
  워커 여러 개가 같은 broker(Redis)에서 task를 나눠 가져가는 게
  celery의 기본 확장 방식. 그래서 "두 개 떠서 문제"라는 가정은
  성립하지 않는다.

  구분    실제 원인                         오해한 원인
  무엇    호스트 celery가 redis 못 찾음(DNS) 도커+호스트 celery 충돌
  관련    .env의 REDIS_HOST 값               (충돌은 애초에 안 남)
  해결    REDIS_HOST=localhost               -

도커에서 celery 삭제 = 에러 해결이 아니라 잉여 서비스 정리.
진짜 해결은 .env의 REDIS_HOST=localhost 한 줄이었다.

[Unit 1 - FastAPI 기초]

-- FastAPI / uvicorn --
FastAPI: Python 비동기 웹 프레임워크. async/await 기반.
uvicorn: ASGI 서버. FastAPI 공식 문서 기준 서버이며 실무 표준.
          (ASGI 서버 종류: uvicorn, hypercorn, daphne 등이 있으나 실무에서는 uvicorn이 사실상 표준)

WSGI vs ASGI:
  WSGI   요청 하나 처리 중 다른 요청 대기 (동기)
  ASGI   요청 대기 중 다른 요청 처리 가능 (비동기)
  FastAPI는 비동기 프레임워크이므로 ASGI 서버(uvicorn)가 필요

비유: FastAPI = 주방, uvicorn = 홀 직원 (요청을 받아 주방에 전달)

설치:
  uv add fastapi uvicorn

실행:
  uvicorn app.main:app --reload
    app.main   app/main.py 파일
    :app       그 파일 안의 app 변수 (FastAPI 인스턴스)
    --reload   코드 변경 시 자동 재시작 (개발용)

-- Request / Response 스키마 (Pydantic 모델) --
API가 받고 반환하는 데이터 구조를 Pydantic BaseModel로 정의.

파일 위치: app/schemas/item.py

코드:
  class ItemCreate(BaseModel):    # 요청 body 스키마
      name: str
      price: float
      is_available: bool = True   # 기본값 있음 → 요청에 없으면 자동으로 True

  class ItemResponse(BaseModel):  # 응답 스키마
      id: int
      name: str
      price: float
      is_available: bool

Q. 기본값 없는 필드는 자동으로 채워지나?
  반대. 기본값 있는 필드가 자동으로 채워진다.
    기본값 없음 (name, price)     → 요청에 반드시 포함해야 함. 없으면 422 Validation Error
    기본값 있음 (is_available)    → 요청에 없으면 기본값으로 자동 채움

response_model:
  @router.post("/items", response_model=ItemResponse)
  엔드포인트가 반환할 스키마를 명시. 두 가지 역할:
    1. 반환 데이터를 스키마에 맞게 필터링 (불필요한 필드 자동 제거)
    2. /docs에 응답 형태 자동 문서화

model_dump():
  Pydantic 모델 인스턴스를 dict로 변환.
  **item.model_dump() → name=..., price=..., is_available=... 형태로 풀어서 전달

fake_db:
  실제 DB 없이 메모리(리스트)에 임시 저장하는 방식.
  서버 재시작 시 데이터 사라짐. Unit 2에서 실제 DB로 교체.

422 Validation Error:
  요청 데이터가 스키마와 맞지 않을 때 FastAPI가 자동으로 반환하는 에러.
  별도 처리 없이 Pydantic이 자동으로 검증.

-- 라우터 분리 구조 --
엔드포인트를 기능별 파일로 나누고 한 곳에서 모아 등록하는 패턴.

구조:
  main.py
    └─ api_router (prefix: /api/v1)
         └─ health_router (tag: health)
              └─ GET /health

파일별 역할:
  main.py              FastAPI 앱 생성, 최상위 라우터 등록만
  api/v1/router.py     v1에 속한 모든 라우터를 한 곳에서 모아 등록
  api/v1/health.py     health 관련 엔드포인트만

코드:
  # api/v1/health.py
  from fastapi import APIRouter
  router = APIRouter()

  @router.get("/health")
  def health_check():
      return {"status": "ok"}

  # api/v1/router.py
  from fastapi import APIRouter
  from app.api.v1.health import router as health_router

  api_router = APIRouter()
  api_router.include_router(health_router, tags=["health"])

  # main.py
  from fastapi import FastAPI
  from app.api.v1.router import api_router

  app = FastAPI()
  app.include_router(api_router, prefix="/api/v1")

확장 시:
  api_router.include_router(health_router, tags=["health"])
  api_router.include_router(user_router, tags=["users"])   ← 추가
  api_router.include_router(auth_router, tags=["auth"])    ← 추가
  main.py는 건드릴 필요 없음

tags: /docs에서 엔드포인트를 그룹으로 묶어 보여주는 용도. 기능적 차이 없음.

Q. __pycache__는 왜 생기나?
  Python이 .py 파일을 실행할 때 바이트코드로 컴파일한 결과를 캐싱해두는 폴더.
  다음 실행 시 소스가 바뀌지 않았으면 컴파일을 건너뛰고 캐시를 사용 → 실행 속도 향상.
  개발자가 직접 건드릴 일 없음. git에 올리지 않도록 .gitignore에 추가.
    __pycache__/
    *.pyc

Q. docker compose가 있는데 왜 uvicorn을 따로 실행하나?
  개발 단계에서는 역할을 나눈다.
    docker compose up          PostgreSQL, Redis 등 인프라
    uv run uvicorn ... --reload FastAPI 앱 (로컬 직접 실행)
  앱을 Docker로 띄우면 코드 변경 시마다 이미지를 새로 빌드해야 해서 불편.
  --reload로 로컬에서 실행하면 저장 즉시 반영.
  나중에 배포 단계(Unit 8)에서 Dockerfile을 만들고 docker-compose.yml에 앱도 추가.

[Unit 0 - 개발 환경 세팅]

-- uv --
Python 패키지 매니저. pip + venv + pyenv를 하나로 합친 것.
Rust로 작성되어 기존 도구들보다 훨씬 빠르다.

명령어:
  uv init --no-readme       프로젝트 초기화
  uv python pin 3.12        Python 버전 고정
  uv add <패키지>            의존성 추가 (pyproject.toml에 자동 기록)
  uv add --dev <패키지>      개발용 의존성 추가
  uv run <명령어>            가상환경 안에서 명령어 실행
  uv sync                   pyproject.toml 기준으로 의존성 동기화

생성 파일:
  pyproject.toml    프로젝트 메타데이터 + 의존성 목록
  .python-version   사용할 Python 버전 기록
  uv.lock           패키지 버전 고정 파일 (커밋 O)
  .venv/            가상환경 폴더 (커밋 X, .gitignore에 추가)

-- pyproject.toml --
Python 프로젝트 표준 설정 파일. setup.py, requirements.txt를 하나로 통합.

섹션:
  [project]             프로젝트 이름, 버전, 의존성 목록
  [dependency-groups]   dev 전용 의존성 (테스트, 린터 등)
  [tool.xxx]            각종 도구 설정 (ruff, pytest 등)

uv add 실행 시 dependencies에 자동 추가됨.

-- Ruff --
Python 린터 + 포매터. flake8 + black + isort를 하나로 합친 것.
Rust로 작성되어 매우 빠르다.

설치:
  uv add --dev ruff   (dependency-groups.dev에 추가됨)

pyproject.toml 설정:
  [tool.ruff]
  line-length = 88          한 줄 최대 길이
  target-version = "py312"  대상 Python 버전

  [tool.ruff.lint]
  select = ["E", "F", "I"]
    E: pycodestyle  - 들여쓰기, 공백 등 스타일 규칙
    F: pyflakes     - 미사용 import, 미정의 변수 등 논리 오류
    I: isort        - import 순서 정렬 (표준 → 서드파티 → 로컬)

  [tool.ruff.format]
  quote-style = "double"    큰따옴표 통일
  indent-style = "space"    들여쓰기 스페이스

명령어:
  uv run ruff check .         린트 검사
  uv run ruff check . --fix   자동 수정
  uv run ruff format .        코드 포맷
  uv run ruff format --check . 포맷 변경사항 확인 (실제 수정 없이)

-- .env --
민감한 값(DB 비밀번호, API 키 등)을 코드에서 분리하는 파일.
git에 올리지 않고 환경마다 따로 관리한다. (.gitignore에 추가)

구조:
  # App
  APP_ENV=development
  APP_DEBUG=true
  SECRET_KEY=super-secret-key-change-in-production

  # Database (PostgreSQL)
  DB_HOST=localhost
  DB_PORT=5432
  DB_NAME=pystack_db
  DB_USER=pystack_user
  DB_PASSWORD=pystack_password

  # Redis
  REDIS_HOST=localhost
  REDIS_PORT=6379

  # MinIO (S3-compatible)
  MINIO_ENDPOINT=localhost:9000
  MINIO_ACCESS_KEY=minioadmin
  MINIO_SECRET_KEY=minioadmin
  MINIO_BUCKET=pystack-bucket

-- pydantic-settings --
.env 파일과 환경변수를 Python 클래스로 읽어오는 라이브러리.
BaseSettings를 상속받아 설정 클래스를 만든다.

설치:
  uv add pydantic-settings

파일 위치: app/core/config.py

코드:
  from pydantic_settings import BaseSettings, SettingsConfigDict

  class Settings(BaseSettings):
      model_config = SettingsConfigDict(
          env_file=".env",
          env_file_encoding="utf-8",
      )

      app_env: str = "development"   # 기본값 있음 → .env에 없어도 OK
      app_debug: bool = False
      secret_key: str                # 기본값 없음 → .env에 반드시 있어야 함

      db_host: str = "localhost"
      db_port: int = 5432
      db_name: str
      db_user: str
      db_password: str

      redis_host: str = "localhost"
      redis_port: int = 6379

      minio_endpoint: str
      minio_access_key: str
      minio_secret_key: str
      minio_bucket: str

  settings = Settings()  # type: ignore[call-arg]

핵심 개념:
  - model_config      .env 파일 경로와 인코딩 설정
  - 기본값 없는 필드   앱 시작 시 .env에 없으면 즉시 에러 → 설정 누락을 조기에 잡음
  - 타입 자동 변환    "5432" → 5432, "true" → True (문자열을 각 타입으로 변환)
  - 대소문자 무관      .env의 DB_HOST가 db_host 필드에 자동 매핑
  - 싱글턴 패턴       모듈 하단에 settings = Settings() 하나만 생성,
                      다른 파일에서 from app.core.config import settings 로 사용

-- docker-compose.yml --
여러 컨테이너를 한 번에 정의하고 실행하는 파일.
서비스를 추가할 때마다 여기에 블록을 추가해간다.

구조:
  services:
    db:
      image: postgres:16
      environment:
        POSTGRES_DB: ${DB_NAME}
        POSTGRES_USER: ${DB_USER}
        POSTGRES_PASSWORD: ${DB_PASSWORD}
      ports:
        - "${DB_PORT}:5432"
      volumes:
        - postgres_data:/var/lib/postgresql/data

    redis:
      image: redis:7
      ports:
        - "${REDIS_PORT}:6379"

  volumes:
    postgres_data:

핵심 개념:
  - services          실행할 컨테이너 목록. 나중에 minio, celery 등 추가
  - image             Docker Hub에서 가져올 이미지. 버전 태그 고정 권장
  - environment       컨테이너 안에 주입할 환경변수. ${변수명}으로 .env 값 참조
  - ports             호스트포트:컨테이너포트. 호스트 쪽만 바꾸면 포트 충돌 해결
  - volumes           컨테이너가 꺼져도 데이터 유지. DB처럼 영속성이 필요한 서비스에 사용
  - Redis 볼륨 없음    캐시 용도라 재시작 시 초기화되어도 무방

명령어:
  docker compose up         포그라운드 실행 (로그 실시간 출력)
  docker compose up -d      백그라운드(데몬) 실행
  docker compose ps         실행 중인 컨테이너 상태 확인
  docker compose down       컨테이너 종료
  docker compose down -v    컨테이너 종료 + 볼륨까지 삭제 (DB 초기화)

컨테이너 접속:
  docker compose exec db psql -U pystack_user -d pystack_db
    exec          실행 중인 컨테이너 안에서 명령어 실행
    db            docker-compose.yml에서 정의한 서비스 이름
    psql          PostgreSQL 클라이언트 CLI
    -U            접속할 유저
    -d            접속할 DB 이름

psql 명령어:
  \conninfo     현재 접속 정보 출력 (호스트, 유저, DB명)
  \q            psql 종료

Q. DB 유저와 DB는 언제 만들어지나?
  docker compose up 실행 시 postgres 공식 이미지가 처음 시작될 때
  environment에 넣어준 값을 읽어 자동으로 생성한다.
    POSTGRES_DB       → 이 이름으로 DB 생성
    POSTGRES_USER     → 이 이름으로 유저 생성
    POSTGRES_PASSWORD → 해당 유저 비밀번호 설정
  이미지 안에 초기화 로직이 내장되어 있어 별도 SQL 실행 불필요.
  volumes가 없으면 컨테이너를 내렸다 올릴 때마다 DB가 초기화된다.
